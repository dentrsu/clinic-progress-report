Treatment Record Verification

How the verification process works in DentRSU Tracker — for students, instructors, and administrators.

DentRSU Tracker

Who is involved?

Student

  • Records treatments in the system
  • Requests verification from instructor
  • Receives proof email after approval
  • Keeps the email as official record

Instructor

  • Receives verification request email
  • Clicks Verify or Reject in the email
  • Can also bulk-verify from the queue
  • Can verify authenticity of proof emails

Admin / Advisor

  • Oversees progress across divisions
  • Can use Standalone Verifier to check any proof email
  • No special action needed in normal flow

What is the Verification Proof Block?

The green box at the bottom of every verification email contains a cryptographic hash — a unique digital fingerprint computed from the exact content of the record at the moment of verification. If any field is changed after verification, the hash will no longer match.

Fields covered by the hash

Section A — From the proof block itself:

Verified At (timestamp) Record ID (UUID) Requirement ID (UUID) Verified By (Instructor ID)

Section B — From the email content table:

RSU Units CDA Units Requirement Type Treatment Name Step Name Area / Teeth Severity (PERIO) Book Number (OPER) Page Number (OPER)

The hash uses HMAC-SHA256 with a server secret key — it cannot be forged without the key. The prefix v2: identifies the algorithm version.

How to Verify an Email's Authenticity

Anyone with instructor or admin access can confirm that a proof email is genuine and that the record has not been altered since verification.

Option A — One Click (Recommended)

  1. Open the verification email.
  2. Find the green Verification Proof block.
  3. Click "Open in Verifier (pre-filled)".
  4. All fields fill automatically.
  5. Click Verify Hash.
  6. Green = authentic. Red = mismatch.

Option B — Manual Entry

  1. Open Verify Email Proof from the home page.
  2. Copy the 5 values from the green proof block into Section A.
  3. Copy the clinical values from the email table into Section B.
  4. Click Verify Hash.
  5. Green = authentic. Red = mismatch or wrong field.
If verification fails: re-check that each value is copied exactly as shown in the email (no extra spaces). A red result means at least one field differs from what was recorded at the time of verification — this may indicate the record was edited after approval.

Frequently Asked Questions

Can I request verification for a new record directly (without saving first)?

Yes. The "Request Verification" button works on both new and existing records. The system saves the record first, then sends the verification email.

What happens if I edit a record after it has been verified?

The record in the system will no longer match the original proof email. If someone runs the verifier against the old email, the hash check will fail — this is intentional and indicates the record was changed post-approval.

The "Open in Verifier" button says the hash is invalid. Why?

This can happen if (1) the record was edited in the database after verification, (2) the email was corrupted or altered, or (3) the proof belongs to a different record. Check all fields against the email carefully.

Is the Standalone Verifier accessible without logging in?

No. The verifier requires an instructor or admin account. The computation itself uses no database, but a system login is required to access the tool.

What does the "v2:" prefix on the hash mean?

It identifies the hash algorithm version (HMAC-SHA256 with content coverage). Older emails without this prefix used a simpler algorithm and can only be checked through the database-assisted admin tool, not the standalone verifier.
